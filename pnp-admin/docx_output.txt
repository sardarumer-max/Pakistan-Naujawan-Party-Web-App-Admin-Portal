Row Level Security (RLS) Policies RLS is database-level security. Even if admin dashboard has a bug, users CANNOT access other users&apos; data. Admin role bypasses all RLS. Table Public Can READ Auth User Can READ Admin Can READ Who Can WRITE profiles ? No Own only ? All Own profile only complaints ? Non-hidden Own only ? All Own complaints jobs ? Verified only Verified only ? All Auth users (post) cvs ? No Own only ? All Own CV only ideas ? Approved only Own only ? All Auth users (submit) idea_votes ? Yes (counts) Own votes ? All Auth users (vote) chat_sessions ? No Own only ? All Own sessions crop_reports ? No Own only ? All Auth users blood_donors ? No Auth users (city/blood) ? All Own record mandi_prices ? Yes ? Yes ? All Admin/scraper only legal_queries ? No Own only ? All Anyone (anon ok) 4.1 How Admin Role Works Users with role = &apos;admin&apos; in profiles table can bypass all RLS restrictions. To make someone an admin: UPDATE public.profiles SET role = &apos;admin&apos; WHERE id = &apos;user-uuid-here&apos;; In admin dashboard code, use the SERVICE ROLE KEY when creating the Supabase client. This automatically bypasses all RLS: const supabase = createClient(SUPABASE_URL, SERVICE_ROLE_KEY); // This query will return ALL complaints, not just public ones const { data } = await supabase.from(&apos;complaints&apos;).select(&apos;*&apos;); 5. Useful Admin SQL Queries These queries are ready to use in Supabase SQL Editor or from admin dashboard code. 6.1 Complaints Dashboard -- Get all critical complaints SELECT id, category, description_raw, district, department, created_at FROM complaints WHERE urgency = &apos;critical&apos; ORDER BY created_at DESC; -- Complaints count by district SELECT district, COUNT(*) as total, SUM(CASE WHEN urgency=&apos;critical&apos; THEN 1 ELSE 0 END) as critical FROM complaints GROUP BY district ORDER BY total DESC; -- Approve a complaint (change status) UPDATE complaints SET status = &apos;in_review&apos; WHERE id = &apos;complaint-uuid&apos;; 6.2 Jobs Moderation -- Get jobs pending review (scam score 0.4-0.7) SELECT id, title, company, ai_scam_score, ai_scam_flags FROM jobs WHERE ai_scam_score BETWEEN 0.4 AND 0.7 ORDER BY ai_scam_score DESC; -- Approve a job UPDATE jobs SET is_verified = true WHERE id = &apos;job-uuid&apos;; -- Reject a job UPDATE jobs SET is_verified = false WHERE id = &apos;job-uuid&apos;; 6.3 Ideas Moderation -- Get pending ideas sorted by AI quality score SELECT id, title, province, ai_quality_score, vote_count FROM ideas WHERE status = &apos;pending&apos; ORDER BY ai_quality_score DESC; -- Approve an idea UPDATE ideas SET status = &apos;approved&apos; WHERE id = &apos;idea-uuid&apos;; -- Get top voted approved ideas SELECT title, vote_count, province, ai_quality_score FROM ideas WHERE status = &apos;approved&apos; ORDER BY vote_count DESC LIMIT 10; 6.4 User Management -- Get all users with their stats SELECT p.id, p.full_name, p.phone, p.province, p.role, p.created_at, COUNT(DISTINCT c.id) as complaints, COUNT(DISTINCT i.id) as ideas FROM profiles p LEFT JOIN complaints c ON c.user_id = p.id LEFT JOIN ideas i ON i.user_id = p.id GROUP BY p.id ORDER BY p.created_at DESC; -- Make user an admin UPDATE profiles SET role = &apos;admin&apos; WHERE id = &apos;user-uuid&apos;; 6.5 Platform Statistics -- Overall platform stats SELECT (SELECT COUNT(*) FROM profiles) as total_users, (SELECT COUNT(*) FROM complaints) as total_complaints, (SELECT COUNT(*) FROM complaints WHERE status=&apos;resolved&apos;) as resolved, (SELECT COUNT(*) FROM ideas WHERE status=&apos;approved&apos;) as approved_ideas, (SELECT COUNT(*) FROM jobs WHERE is_verified=true) as active_jobs, (SELECT COUNT(*) FROM blood_donors WHERE is_available=true) as blood_donors;
